# 司南榜报名

## 来源

- 技术方案：`docs/tech-design/sinan-sign.md`
- TODO: 补充原始 PRD 链接。

## 需求背景

司南榜需要在榜单开始前收集机构和医生的报名意向。缺少统一报名入口，商家/医师无法查看自己机构及机构下医生已报名哪些项目，也无法完成提报或退报。目标报名商户约 3500 家。

## 目标

- 为机构榜和医生榜提供统一报名入口
- 支持提报、退报、报名结果分页查询、可选提报项目查询、机构下医生列表查询
- 支持报名窗口期卡控，窗口期结束后不允许提报和退报
- 报名数据持久化到 MySQL，供后续筛选、材料提报和进度查看关联使用
- 医生榜报名时支持选择机构下在线和不在线医生

## 影响领域

| 领域 | 影响 |
| --- | --- |
| sign（报名） | 核心变更：新增报名聚合、4 个 Facade 接口、t_sign 表 |
| material（材料） | 后续依赖报名记录和报名状态 |
| review（评审） | 后续依赖报名状态 |
| score（打分） | 后续依赖报名主体 |
| report（报告） | 后续依赖报名结果 |

## 提炼出的业务规则

- 报名记录粒度为"主体 x 场景 x 提报项目"，同一 `场景+主体+项目` 只存在一行记录
- 状态通过 SIGNED / CANCELLED 流转，不做物理删除
- 机构榜报名主体为 shopId，医生榜为 techId；indexShopId 冗余机构 ID 支撑机构维度查询
- 每个报名主体最多提报 2 个项目
- 医生榜同一机构最多报名 10 个医生
- 窗口期外提报和退报均被拦截（fail closed）
- 提报时：入参项目不存在则新增，已退报则恢复，已提报则幂等跳过
- 退报时：入参项目已提报则退报，已退报或不存在则跳过
- 分页查询按机构维度（indexShopId），不支持按单个 subjectId 精确查询
- 可选提报项目仅返回项目 code 列表，不返回窗口期状态
- 医生列表通过 DoctorAcl 防腐层查询（当前 Mock 实现）
- 窗口期和可选项目通过 SignConfigRepository 读取 Lion 配置（当前 Mock 实现）

## 流程变化

1. 新增报名操作：BFF → SignFacade.operateSign → 校验窗口期/项目可选性/数量限制 → 操作 t_sign
2. 新增报名分页查询：BFF → SignFacade.querySignPage → 按 indexShopId 分页查询
3. 新增可选项目查询：BFF → SignFacade.querySelectableProjects → SignConfigRepository
4. 新增医生列表查询：BFF → SignFacade.queryShopDoctors → DoctorAcl

## 验收标准

| 场景 | 预期 |
| --- | --- |
| 机构榜提报 | shopId 提报成功，写入 SIGNED 记录 |
| 医生榜提报 | techId 提报成功，不超 2 项目/10 医生限制 |
| 重复提报幂等 | 已有 SIGNED 记录时跳过，不产生重复行 |
| 退报后重新提报 | CANCELLED 恢复为 SIGNED |
| 退报 | SIGNED 更新为 CANCELLED |
| 窗口期后提报/退报 | 均被拦截，返回业务异常 |
| 超项目数限制 | 第 3 个项目时返回参数非法 |
| 超医生数限制 | 第 11 个医生时返回参数非法 |
| 分页查询 | 按机构 ID 分页返回报名记录 |
| 可选项目查询 | 返回当前场景可选项目 code 列表 |
